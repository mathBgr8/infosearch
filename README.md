# Краулер для starwars.fandom.com и построение индекса по полученным документам

## Описание

Скрипт crawl_50000.py предназначен для массовой обкачки сайта starwars.fandom.com с извлечением 50000 страниц через MediaWiki API. Скрипт сохраняет каждую страницу в отдельный JSON-файл.

## Требования

- Python 3.7+
- Установленные зависимости:
  - requests

Установка зависимостей:
```bash
pip install requests
```

## Использование

```bash
python crawl_50000.py
```

## Как это работает

### 1. Загрузка robots.txt
Скрипт загружает и парсит siteinfo/robots.txt для определения запрещенных путей. Проверяет правила для User-Agent: *.

### 2. Извлечение URL из sitemap
- Читает siteinfo/indexes/sitemap-newsitemapxml-index.xml
- Выбирает все sitemap-файлы с NS_0 (основное вики)
- Извлекает URL, lastmod, priority из каждого sitemap
- Собирает до 50000 URL (берет первые 4-5 файлов NS_0)

### 3. Фильтрация по robots.txt
Каждый URL проверяется через is_allowed(). Запрещенные пути (Special:, User:, Template: и др.) отфильтровываются.

### 4. Checkpoint система
- При старте загружает checkpoint.txt со списком уже обработанных URL
- Пропускает обработанные URL
- Сохраняет прогресс каждые 10 страниц
- При прерывании (выключение или Ctrl+C) сохраняет чекпоинт

### 5. Обкачка через MediaWiki API
Для каждого URL:
- Извлекает название страницы из URL
- Вызывает API: action=query&titles={title}&prop=revisions&rvprop=content
- Добавляет crawl-delay 1 секунда (требование robots.txt)
- Обрабатывает ошибки (404, 500, API ошибки)

### 6. Сохранение результатов
Каждая успешно загруженная страница сохраняется в pages/{pageid}.json:

```json
{
  "url": "https://starwars.fandom.com/wiki/Page_Title",
  "title": "Page Title",
  "pageid": 12345,
  "content": "{{wikitext content}}",
  "length": 1234,
  "lastmod": "2024-01-15T10:30:00Z",
  "priority": 0.8
}
```

### 7. Логирование
Все действия логируются в crawl_log.txt с временными метками:
- Прогресс (обработано N/M)
- Статистика (успешно, ошибок)
- Ошибки и предупреждения
- Итоговая статистика

## Возобновление после прерывания

Если скрипт был прерван (Ctrl+C или ошибка):
1. Запустите его снова
2. Он автоматически загрузит checkpoint.txt
3. Пропустит уже обработанные URL
4. Продолжит с места остановки

## Статистика

В конце работы выводится:
- Всего обработано
- Успешно загружено
- Ошибок
- Общее время
- Среднее время на страницу

## Обработка ошибок

Скрипт пропускает:
- Страницы с HTTP ошибками (404, 500, etc.)
- Отсутствующие страницы (pageid = -1)
- API ошибки (неверный JSON, таймауты)
- Ошибки сохранения файлов

Ошибки логируются, но не останавливают краулер.

## Ожидаемое время

При ~1.5 секунды на страницу:
- 1000 страниц: ~25 минут
- 10000 страниц: ~4 часов
- 50000 страниц: ~21 час

# Булевый поиск со стеммингом и анализом закона Ципфа

Система для построения булевого индекса и поиска по коллекции из 40000+ документов с поддержкой русского и английского языков, стеммингом и анализом распределения частот слов (закон Ципфа).

## Архитектура

### C++ компоненты (ядро системы)

1. **tokenizer.h/cpp** - Токенизация текста
   - Поддержка русского и английского языков
   - Нормализация (приведение к нижнему регистру)
   - Фильтрация токенов (удаление чисел, коротких слов)

2. **stemmer.h/cpp** - Стемминг
   - Стеммер Портера
   - Поддержка английских суффиксов

3. **indexer.h/cpp** - Построение и хранение индекса
   - Обратный индекс: термин → список документов
   - Булевые операции: AND (пересечение), OR (объединение), NOT (разность)
   - Сериализация/десериализация в бинарный формат
   - Статистика по частотам терминов

4. **main.cpp** - Главная программа
   - Построение индекса из папки с документами
   - Интерактивный поиск
   - Загрузка/сохранение индекса

### Python компоненты

1. **convert_json.py** - Конвертация JSON в текстовый формат
   - Преобразует JSON файлы из папки pages/ текстовый формат
   - Извлекает метаданные и содержимое

2. **zipf_analysis.py** - Анализ закона Ципфа
   - Токенизация и стемминг
   - Построение графика распределения частот
   - Оценка коэффициента Ципфа
   - Визуализация в log-log масштабе

## Установка и сборка

### Требования

- C++17 совместимый компилятор (g++, clang++, MSVC)
- Python 3.7+ (для анализа Ципфа)
- Библиотеки Python: matplotlib, numpy

### Сборка C++ проекта

```bash
# Linux/macOS
make
```

# Или вручную
```bash
g++ -std=c++17 -O3 -Wall -Wextra -Wpedantic -o bool_search \
    main.cpp indexer.cpp tokenizer.cpp stemmer.cpp -lstdc++fs
```

Для Windows (MinGW):
```bash
g++ -std=c++17 -O3 -Wall -Wextra -Wpedantic -o bool_search.exe \
    main.cpp indexer.cpp tokenizer.cpp stemmer.cpp
```

## Использование

### 1. Конвертация JSON документов

```bash
python convert_json.py pages/ txt_docs/
```

Это создаст текстовые файлы в папке txt_docs/ в формате:
```
id
url
title
length

содержимое документа...
```

### 2. Построение индекса

```bash
# Строим индекс из папки с документами
./bool_search txt_docs/ index.bin

# Или загружаем существующий индекс и переходим к поиску
./bool_search index.bin --search
```

### 3. Интерактивный поиск

После построения/загрузки индекса откроется интерактивный режим:

```
=== БУЛЕВЫЙ ПОИСК ===
Введите запрос (слова через пробел = AND, используйте OR, NOT для сложных запросов):
Для выхода введите exit

**Формат запросов:**
- слово1 слово2 - AND (оба слова должны присутствовать)
- слово1 OR слово2 - OR (хотя бы одно слово)
- NOT слово - NOT (исключить документы с этим словом)
- Комбинации: слово1 AND NOT слово2

### 4. Анализ закона Ципфа

```bash
python zipf_analysis.py pages/ (тут можно указать число, меньшее размера выборки)
```

**Результаты:**
- График zipf_analysis.png с двумя панелями:
  - Левый: log-log график частот (проверка закона Ципфа)
  - Правый: Топ-50 самых частых слов
- Файл zipf_stats.json с подробной статистикой
- Вывод в консоль коэффициента Ципфа

## Формат индекса

Бинарный файл index.bin содержит:
1. Количество документов (size_t)
2. Количество терминов (uint64_t)
3. Для каждого термина:
   - Длина термина (uint32_t)
   - Термин в UTF-8
   - Размер posting list (uint32_t)
   - Posting list (массив int)

## Оптимизации для 40000+ документов

1. **Память:**
   - Posting lists хранятся сжато (отсортированные int массивы)
   - Используется unordered_map(myhashmap.h) для быстрого доступа к терминам

2. **Скорость поиска:**
   - Intersect использует линейный алгоритм за O(n+m)
   - Posting lists предварительно отсортированы
   - Кэширование частых запросов (опционально)

3. **Стемминг:**
   - Простой стеммер Портера работает быстро (<1 мин)

## Демонстрация закона Ципфа

Закон Ципфа гласит: частота слова обратно пропорциональна его рангу.

f(r) ≈ C/r^s

где:
- f(r) - частота слова с рангом r
- C - константа
- s - коэффициент Ципфа (обычно ≈ 1 для естественных языков)

**Визуализация:**
- Построение log(f) vs log(r) должно давать прямую с наклоном ≈ -1
- Наш анализ автоматически вычисляет коэффициент и строит график

## Тестирование

### Быстрый тест на небольшой выборке

```bash
# Конвертируем первые 1000 документов
python convert_json.py pages/ test_docs/ --sample 1000

# Строим индекс
./bool_search test_docs/ test_index.bin

# Проверяем поиск
./bool_search test_index.bin --search
```

### Полный тест на всех документах

```bash
# Конвертация всех ~45000 документов
python convert_json.py pages/ all_docs/

# Построение полного индекса (может занять несколько минут)
./bool_search all_docs/ full_index.bin

# Анализ Ципфа на полной коллекции
python zipf_analysis.py pages/
```

## Ожидаемые результаты

Для коллекции из 40000+ документов на английском:

- **Уникальных терминов после стемминга:** ~50,000-100,000
- **Размер индекса:** ~100-500 MB (в зависимости от среднего размера документа)
- **Время поиска:** < 10 мс для простых запросов
- **Коэффициент Ципфа:** -0.9 до -1.1
